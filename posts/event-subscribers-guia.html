¿Qué son los Event Subscribers?

Los Event Subscribers son el mecanismo principal para extender la funcionalidad de Business Central sin modificar el código fuente. Permiten "escuchar" cuando ocurre algo en el sistema y ejecutar tu lógica personalizada.

Tipos de eventos

1. Integration Events

Los más comunes. Son publicados por Microsoft o por otras extensiones para permitir que terceros extiendan la lógica:

// El publisher define el evento
[IntegrationEvent(false, false)]
local procedure OnBeforePostSalesDoc(
    var SalesHeader: Record "Sales Header";
    var IsHandled: Boolean)
begin
end;

2. Business Events

Indican que algo de negocio ha ocurrido. No se puede modificar el comportamiento, solo reaccionar:

[BusinessEvent(false)]
local procedure OnAfterCustomerCreated(
    CustomerNo: Code[20])
begin
end;

3. Trigger Events

Se generan automáticamente para los triggers de tabla (OnInsert, OnModify, OnDelete, OnValidate). No necesitas declararlos, BC los publica automáticamente:

// Suscribirse al OnAfterValidate del campo Name de Customer
[EventSubscriber(ObjectType::Table, Database::Customer,
    'OnAfterValidateEvent', 'Name', false, false)]
local procedure CustomerNameValidated(
    var Rec: Record Customer;
    var xRec: Record Customer;
    CurrFieldNo: Integer)
begin
    // Tu lógica al cambiar el nombre del cliente
end;

Estructura del atributo EventSubscriber

[EventSubscriber(
    ObjectType::Codeunit,       // Tipo de objeto que publica
    Codeunit::"Sales-Post",     // Objeto específico
    'OnBeforePostSalesDoc',     // Nombre del evento
    '',                         // Element name (para triggers de campo)
    false,                      // SkipOnMissingLicense
    false                       // SkipOnMissingPermission
)]

Buenas prácticas

  • Un codeunit por área funcional: No mezcles suscriptores de ventas con los de compras
  • Nombra claramente: El nombre del procedure debe indicar qué hace, no a qué se suscribe
  • Evita lógica pesada en suscriptores de eventos frecuentes (OnAfterGetRecord, por ejemplo)
  • Usa el patrón IsHandled cuando tu evento lo soporte
  • Documenta por qué te suscribes a cada evento
  • Testea siempre que el evento se dispara cuando esperas

Errores comunes

  • Olvidar que los parámetros var modifican el registro original
  • Crear bucles infinitos al modificar un registro dentro de su propio OnModify
  • No manejar errores dentro del suscriptor (un error en un subscriber rompe todo el proceso)
  • Suscribirse al evento equivocado (OnBefore vs OnAfter)
